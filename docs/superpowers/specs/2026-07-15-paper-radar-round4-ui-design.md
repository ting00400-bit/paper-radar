# Paper Radar Round 4 前端版面設計

**日期：** 2026-07-15（Asia/Taipei）

**狀態：** 已由 Ting 核准方案 A，待書面複核後進入實作計畫

**範圍：** `site/index.html`、`site/style.css`、`site/app.js` 與前端測試

## 1. 目標

把 Paper Radar 從一次鋪滿 backlog 的長頁面，調整成適合手機快速判斷「今天先看什麼」的單欄雷達，同時保留現有 D1 動作與 paper pipeline 語意。

成功條件：

- 手機首次開啟先看到本週新文，不必先滑過全部未看文章。
- 首頁每次最多建立 25 張論文卡，避免一次渲染 100 張以上。
- 一張卡能依序看懂推薦分數、論文內容、推薦原因與主要操作。
- `已看`、`整理筆記`、投票與 `上傳 PDF` 保持在主要操作區。
- 所有既有動作仍寫入相同 D1 欄位，不需要 migration。
- 鍵盤、螢幕閱讀器與手機觸控操作都有基本可用性。

## 2. 非目標

這一輪不處理：

- 不更換現有藍綠色品牌配色。
- 不引入 React、Vue、virtual list 或任何新套件。
- 不改 `actions`、`action_log` 的 schema、欄位語意或 Worker API。
- 不合併 `star`、`deepread`、`content`，也不改 paper pipeline 的工作模式。
- 不處理 PublicationType、OA landing page 分流或 Worker PDF 驗證；這些屬於後續資料品質與安全工作。
- 不重做同步狀態 dashboard 與 PRPM 訓練邏輯。

## 3. 導航與預設範圍

論文範圍採四個明確分頁：

1. `本週新文`
2. `全部未看`
3. `已看`
4. `同步狀態`

`本週新文` 使用現有 `isNew === true`，並排除已標記 `seen` 的論文。主題、搜尋與 badge 篩選仍會套用在這個結果上。

首次開啟時：

- 視窗寬度小於 600 px：預設 `本週新文`。
- 其他尺寸：預設 `全部未看`。
- 使用者切換分頁後，沿用現有 `LS_FILT` 儲存選擇；之後開啟不因裝置寬度覆蓋使用者選擇。
- 舊版只有 `unseen`／`seen`／`sync` 的儲存值可直接沿用，不需要另做資料遷移。

若本週沒有新文，顯示清楚的空狀態與「查看全部未看」按鈕，不自動切換分頁。

## 4. 分批渲染

論文分頁一次最多渲染 25 張卡片：

- 初次渲染：`visibleCount = 25`。
- 還有結果時，列表底部顯示 `載入更多`。
- 每次點擊再增加 25 篇。
- 切換論文分頁、主題、badge、搜尋或排序時，重設為 25 篇。
- `已看` 分頁仍沿用目前的看過時間排序規則。
- `同步狀態` dashboard 維持現況，不納入這個分頁機制。
- Footer 顯示「顯示 X / Y 篇」，讓使用者知道尚有多少內容。

實作只用陣列 `slice(0, visibleCount)`；資料量未達需要 virtual list 的程度。

## 5. 單欄緊湊卡片

採核准的方案 A，保留一欄卡片，不改成桌面雙欄或大型 hero card。

每張卡片由上到下排列：

1. **標題列**
   - PRPM `score` 是標題旁的小 badge，不再占滿一整列。
   - 有 `p.url` 時，標題是開啟 PubMed／原始來源的連結，使用新分頁。
   - 沒有 `p.url` 時，標題維持純文字，不產生無效連結。
   - 複製標題按鈕留在標題旁，維持現有功能。
2. **來源資訊**
   - 期刊／來源、日期與前三位作者維持次要文字。
3. **標記列**
   - 保留 NEW、OA、機構訂閱、主題標籤與探索標記。
   - 有 `kw_score` 時，以低強度文字顯示 `Keyword N`。
4. **推薦原因**
   - 直接顯示最多三個 `why`，格式為簡短的 `標籤 +權重`。
   - 沒有 `why` 時不建立空容器。
5. **摘要**
   - 只有存在 `abstract` 時才顯示「摘要」按鈕。
   - 使用原生 `<button>`，具備 `aria-expanded` 與 `aria-controls`。
   - 展開後保留摘要複製按鈕。
6. **主要操作列**
   - `已看`
   - `整理筆記`
   - `上傳 PDF`
   - 👍／😐／👎
7. **更多整理方式**
   - 使用原生 `<details>`。
   - 內含 `品質評讀` 與 `內容整理`。

## 6. 動作與資料相容性

只改呈現位置，不改動作語意：

| 顯示名稱 | 既有 key | 行為 |
|---|---|---|
| 已看 | `seen` | 維持現有淡出與未看／已看分流 |
| 整理筆記 | `star` | 維持既有筆記請求 |
| 品質評讀 | `deepread` | 放進更多整理方式，仍隱含標記已看 |
| 內容整理 | `content` | 放進更多整理方式，仍隱含標記已看 |
| 評價 | `vote` | 維持 `up`／`neutral`／`down` |
| 上傳 PDF | 既有 upload route | 保持主要顯示，成功後仍顯示已上傳狀態 |

`pending_ops`、離線重試、跨分頁鎖與 PRPM event context 均沿用現有程式，不因版面改動重寫。

## 7. 無障礙與手機操作

- 主題切換從可點擊 `div` 改為原生 `<button aria-pressed>`。
- 搜尋、檔案、標題、DOI 與排序控制都有可讀的 `<label>` 或 `aria-label`。
- 摘要按鈕具備正確的 `aria-expanded`。
- 所有 icon-only 投票按鈕都有明確 `aria-label`。
- 所有互動元件保留可見的 `:focus-visible` 樣式。
- 手機寬度下，主要操作按鈕提供至少接近 44 px 的觸控高度；必要時自然換行，不產生水平捲動。
- `更多整理方式` 使用原生 `<details>`／`<summary>`，不自製鍵盤互動。
- 次要文字顏色調整到至少符合一般文字 4.5:1 的對比目標。

## 8. 錯誤與邊界狀態

- `p.url`、`abstract`、`kw_score`、`why` 或 `explore` 缺少時，卡片仍正常顯示。
- 本週新文、搜尋或篩選結果為空時，顯示對應空狀態，不留下空白頁。
- `載入更多` 只在 `visibleCount < filteredPapers.length` 時顯示。
- 動作同步失敗仍由既有未同步提示與 `pending_ops` 處理。
- 上傳失敗仍沿用既有訊息，不因按鈕移位改變錯誤流程。
- 本 session 新標記已看的卡片仍保留 2.5 秒，讓使用者有時間補按第二個操作。

## 9. 測試策略

依 TDD 實作，先新增失敗測試，再改正式程式。

Node 測試至少涵蓋：

- 手機首次使用預設 `weekly`，桌面首次使用預設 `unseen`。
- 已儲存的分頁不會被視窗寬度覆蓋。
- `weekly` 只回傳 `isNew` 且尚未看過的文章。
- 每批最多 25 篇，載入更多後增加 25 篇。
- 切換分頁或篩選會重設分頁數量。
- score badge 與標題來源連結的 HTML 結構。
- 有摘要才顯示可存取的摘要按鈕。
- 主要操作包含已看、整理筆記、投票與上傳 PDF。
- 品質評讀與內容整理位於更多區，但仍使用原 key。
- 主題按鈕、輸入 label 與 `aria` 屬性存在。

完整驗證：

- `node --test tests/test_worker.js tests/test_site.js`
- `python -X utf8 -m pytest -q`
- 桌面 1440 × 900 與手機 390 × 844 的 Playwright 實際頁面檢查。
- 檢查手機沒有水平溢位、主控台錯誤或失敗請求。
- 檢查分批渲染後 DOM 卡片數不超過 25。

## 10. 部署與回復

- 這一輪沒有 D1 migration。
- 本機測試與瀏覽器驗證通過後，再同步 GitHub 與 NAS。
- Cloudflare Pages 部署前仍需核對 NAS runtime dirty files，不能覆蓋 `interest_model.json`、log 或其他執行狀態。
- 若線上驗證失敗，可直接重新部署上一個 Pages commit；D1 與 pipeline 資料不受影響。
- `paper_sync.py done` 不屬於本輪驗證，不會執行。
